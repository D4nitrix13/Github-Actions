# Explicación del fichero de configuración `/.github/workflows/server.yml`

Ruta: [`/.github/workflows/server.yml`](/.github/workflows/server.yml "/.github/workflows/server.yml")

```yaml
name: Server

on:
  push:
    branches:
      - master

jobs:
  hello_world:
    runs-on: ubuntu-18.04

    steps:
      - name: Echo the message
        run: |
          echo "Hello World"
```

---

## Visión general

Este workflow define un pipeline de GitHub Actions que se ejecuta cuando hay un `push` a la rama `master`. Dentro del pipeline existe un job (`hello_world`) que corre en un runner Ubuntu y ejecuta un único step que imprime `Hello World`.

---

## 1) `name`

```yaml
name: Server
```

**Qué es:** nombre visible del workflow en la pestaña **Actions**.
**Para qué sirve:** identifica el workflow en la UI, historial y logs.
**Por qué va aquí:** es metadato global del workflow (nivel raíz).

---

## 2) `on`

```yaml
on:
  push:
```

**Qué es:** bloque de eventos que disparan el workflow.
**Para qué sirve:** define *cuándo* debe ejecutarse el pipeline.
**Por qué va aquí:** aplica al workflow completo, no a un job específico.

### 2.1) `push`

```yaml
on:
  push:
```

**Qué significa:** corre cuando se hace `git push` al repositorio remoto.
**Qué incluye el evento:** información como commit, autor, rama, timestamp.

### 2.2) `branches`

```yaml
on:
  push:
    branches:
      - master
```

**Qué es:** filtro del evento `push` por rama.
**Para qué sirve:** evita ejecutar el pipeline para pushes en otras ramas.
**Qué significa `- master`:** solo se ejecuta si el push ocurre en `master`.

---

## 3) `jobs`

```yaml
jobs:
```

**Qué es:** contenedor de trabajos del workflow.
**Para qué sirve:** permite definir uno o varios jobs (paralelos o con dependencias).
**Por qué va aquí:** es el “cuerpo” de ejecución del workflow.

### 3.1) `hello_world`

```yaml
jobs:
  hello_world:
```

**Qué es:** ID del job.
**Para qué sirve:** referencia interna para logs y dependencias (`needs`).
**Nota:** este identificador no tiene espacios y suele ser `snake_case`.

---

## 4) `runs-on`

```yaml
runs-on: ubuntu-18.04
```

**Qué es:** define el runner (SO/imagen) donde se ejecuta el job.
**Para qué sirve:** GitHub necesita saber qué entorno provisionar (Ubuntu/Windows/macOS).
**Recomendación actual:** usar `ubuntu-latest` salvo que necesites una versión fija.

---

## 5) `steps`

```yaml
steps:
```

**Qué es:** lista ordenada de pasos del job.
**Para qué sirve:** ejecutar comandos o acciones (“uses”) en secuencia dentro del mismo runner.
**Detalle técnico:** todos los steps comparten el mismo filesystem y contexto del job.

### 5.1) `- name`

```yaml
- name: Echo the message
```

**Qué es:** etiqueta descriptiva del step.
**Para qué sirve:** mejora la lectura de logs y facilita ubicar fallos.
**Nota:** no cambia el comportamiento, solo documentación/observabilidad.

### 5.2) `run`

```yaml
run: |
  echo "Hello World"
```

**Qué es:** ejecuta comandos de shell en el runner.
**Para qué sirve:** correr scripts, instalar dependencias, ejecutar tests, builds, etc.

#### Uso de `|` (bloque multilínea)

Permite escribir múltiples comandos:

```yaml
run: |
  echo "Linea 1"
  echo "Linea 2"
```

Sin `|`, normalmente usas una sola línea:

```yaml
run: echo "Hello World"
```

---

## Flujo de ejecución

Cuando haces:

```bash
git push origin master
```

Sucede:

1. GitHub recibe el evento `push`.
2. Aplica el filtro `branches: master`.
3. Provisiona un runner `ubuntu-18.04`.
4. Ejecuta el job `hello_world`.
5. Ejecuta los `steps` en orden.
6. Imprime `Hello World` en los logs.
